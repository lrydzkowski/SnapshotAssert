# Design: revert-empty-value-whitespace

## Context

The `2026-07-12-fix-code-review-findings` change (decision D7) made `VerifyTextWriter` suppress the separator space after a property colon and the newline-plus-indent inside arrays whenever the written value is an empty string. The stated goal was output with no lines ending in whitespace, so trim-on-save editors cannot corrupt `.verified.txt` files. The cost turned out to be larger than the benefit: every pre-existing snapshot containing an empty string value — most importantly snapshots generated by real Verify 31.x in projects migrating to SnapshotAssert — now fails strict comparison (`ApiKey: ` vs `ApiKey:`), contradicting the `snapshot-serialization` requirement of byte-for-byte Verify 31.x compatibility.

D7 also introduced a second, unrelated protection: property names containing `\n`/`\r` are escaped so a name cannot break line structure. That part has no snapshot-compat pressure and stays.

The pre-D7 writer behavior is fully recoverable from git (`git diff 06f43b2 306ce36 -- '*VerifyTextWriter.cs'`).

## Goals / Non-Goals

**Goals:**

- Empty string values render exactly as Verify 31.x renders them: `Name: ` in object context; normal newline plus indentation in array context.
- Verify-era snapshots verify without edits; the byte-for-byte compatibility requirement holds again for empty strings.
- Spec, writer, and tests stay consistent with one another.

**Non-Goals:**

- Reverting property-name escaping (`EscapeName`) or the dictionary-key-newline scenario.
- Changing the top-level `emptyString` sentinel for `Verify(string.Empty)` in `InnerVerifier`.
- Any comparison-side whitespace tolerance in `InnerVerifier` — comparison stays strict byte equality.
- Guarding against editors trimming trailing whitespace (accepted trade-off; mitigated by documentation only).

## Decisions

### D1: Exact mechanical revert of the empty-value special case in `VerifyTextWriter`

- `WriteRaw(string value)` calls `BeginValue()` unconditionally instead of `BeginValue(value.Length == 0)`.
- `BeginValue` loses its `isEmptyValue` parameter: the pending-property branch always appends the separator space; the array branch always calls `AppendNewLineAndIndent`.
- Everything else in the writer — `EscapeName`, the multi-line `WriteString` branch, scope handling — is untouched.
- Alternative considered: keeping the parameter but defaulting it off. Rejected: dead code with no caller.

Resulting Verify-compatible renderings (these become the test expectations):

- Empty property value: `{\n  Name: \n}` — the `Name: ` line ends in one space.
- `["a", "", "b"]`: `[\n  a,\n  ,\n  b\n]` — the middle element's line is `  ,` (indentation, then the next element's separator comma; incidentally no trailing whitespace).
- `["a", ""]`: `[\n  a,\n  \n]` — the final element's line is exactly the two-space indentation.

### D2: Spec restructuring in `snapshot-serialization`

- REMOVED: "Whitespace-safe output format" — its umbrella guarantee (no line ends in whitespace) is withdrawn.
- ADDED: "Single-line property names" — carries forward only the name-escaping requirement and the dictionary-key-newline scenario, unchanged in behavior.
- MODIFIED: "Verify-compatible default text format" — gains explicit empty-string scenarios locking the trailing-space rendering, so this compatibility point can never silently drift again.
- Alternative considered: keeping a weakened "Whitespace-safe" requirement covering only non-empty values. Rejected: a value like `"x "` legitimately renders with trailing whitespace, so the guarantee would be false as stated in any general form.

### D3: Downstream safety of reintroduced trailing spaces

`FixNewlines` and `ApplyScrubbers` only normalize line endings and substitute tokens; nothing in the pipeline trims line-trailing whitespace, so the writer's output reaches the comparison and the `.received.txt` file byte-intact. Verified by inspection of `StringBuilderScrubbingExtensions`.

### D4: Versioning and consumer guidance

- Bump `Version` to 0.2.0: rendering changes relative to the current 0.1.0 behavior, and any snapshot approved against the post-D7 format must be re-approved.
- README gains a short note recommending the Verify-ecosystem `.editorconfig` stanza for snapshot files (`trim_trailing_whitespace = false`, `insert_final_newline = false` for `*.verified.txt` / `*.received.txt`), which is the standard mitigation for the editor-trim risk that motivated D7.

## Risks / Trade-offs

- [Editors with trim-on-save corrupt verified files containing empty values] → Accepted deliberately; mitigated by the README `.editorconfig` guidance. This is the same exposure every Verify user has.
- [Consumers who already approved post-D7 snapshots break on upgrade] → Called out as **BREAKING** in the proposal; re-approving the affected files is a per-line trailing-whitespace-insertion the diff tool makes obvious. The known consumer set is small and pre-1.0.
- [Repo-owned snapshots drift] → Checked: no `.verified.txt` in this repo contains an empty-value line; only unit-test expectation strings change.

## Migration Plan

Single commit; no deployment concerns beyond the version bump. Rollback is `git revert`.
